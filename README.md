# ระบบร้านค้าออนไลน์ขายสินค้าลิขสิทธิ์แท้(Merchandise Online Shop)

## 1. บทนำ (Introduction)
โครงการนี้เป็นการพัฒนาแพลตฟอร์มดิจิทัลในรูปแบบเว็บแอปพลิเคชันสำหรับซื้อขายสินค้า Merchandise (ของสะสม, เสื้อผ้า, และอุปกรณ์เชียร์ของศิลปิน) เพื่ออำนวยความสะดวกให้แก่กลุ่มแฟนคลับในการเข้าถึงสินค้าลิขสิทธิ์แท้อย่างรวดเร็ว ปลอดภัย และมีระบบการจัดการคลังสินค้าที่มีประสิทธิภาพสำหรับผู้ขาย

## 2. ขอบเขตของระบบ (System Scope)
ระบบแบ่งผู้ใช้งานออกเป็น 2 กลุ่มหลัก โดยมีขอบเขตการทำงานดังนี้:

### 2.1 ฝั่งลูกค้า/ผู้ใช้งานทั่วไป (Customer Side)
* **ระบบสมาชิก:** สามารถสมัครสมาชิก, เข้าสู่ระบบ, และแก้ไขข้อมูลส่วนตัวได้
* **ระบบเลือกชมสินค้า:** สามารถค้นหาสินค้า แยกประเภทตามหมวดหมู่ (เช่นเสื้อ, พวงกุญแจ, แท่งไฟ) และดูจำนวนสินค้าคงเหลือได้
* **ระบบตะกร้าสินค้า:** สามารถเพิ่มสินค้าลงตะกร้า ปรับเพิ่ม-ลดจำนวน และคำนวณราคาสินค้าทั้งหมดได้
* **ระบบชำระเงินและแจ้งโอน:** รองรับการชำระเงินด้วยการอัปโหลดสลิปธนาคาร หรือระบบ QR Code
* **ระบบติดตามสถานะ:** ลูกค้าสามารถตรวจสอบสถานะการจัดส่งและเลขพัสดุ (Tracking Number) ได้

### 2.2 ฝั่งผู้ดูแลระบบ (Admin Side)
* **ระบบจัดการสินค้า (CRUD):** สามารถเพิ่ม ลบ แก้ไข ข้อมูลสินค้า รูปภาพ ราคา และจำนวนสินค้าในสต็อกได้
* **ระบบจัดการคำสั่งซื้อ:** ตรวจสอบประวัติการสั่งซื้อ ตรวจสอบสลิปการโอนเงิน และอัปเดตสถานะจัดส่งสินค้า

## 3. ความต้องการของระบบ (System Requirements)
### 3.1 ความต้องการเชิงฟังก์ชัน (Functional Requirements)
* **FR-1:** ระบบต้องแสดงสินค้าที่อัปเดตล่าสุดบนหน้าแรก
* **FR-2:** ระบบต้องตัดจำนวนสินค้าในสต็อกทันทีหลังจากที่ลูกค้าได้รับการยืนยันคำสั่งซื้อ
* **FR-3:** ระบบต้องส่งอีเมลแจ้งเตือนใบเสร็จรับเงินเมื่อการชำระเงินสำเร็จ

### 3.2 ความต้องการที่ไม่ได้เกี่ยวข้องกับฟังก์ชัน (Non-Functional Requirements)
* **Security (ความปลอดภัย):** รหัสผ่านของผู้ใช้งานทุกคนต้องผ่านการเข้ารหัส (Encryption) ก่อนบันทึกลงฐานข้อมูล และใช้โปรโตคอล HTTPS เพื่อความปลอดภัยในการส่งข้อมูล
* **Performance (ประสิทธิภาพ):** หน้าเว็บต้องมีความเร็วในการแสดงผล (Page Load Time) 
* **Availability (ความพร้อมใช้งาน):** ระบบต้องมีความเสถียร สามารถใช้งานได้ต่อเนื่อง 24 ชั่วโมง โดยมีอัตราความผิดพลาดต่ำ

## 4. สถาปัตยกรรมของระบบ (System Architecture)
ระบบนี้ใช้สถาปัตยกรรมแบบ 3-Tier Architecture ที่มีการแยกส่วนการทำงานอย่างชัดเจนเพื่อความง่ายในการขยายตัวและบำรุงรักษา

```mermaid
graph TD
    %% Users
    Customer[ลูกค้า / ลูกค้าแฟนคลับ]
    Admin[ผู้ดูแลระบบ / ร้านค้า]

    %% Frontend
    subgraph Frontend_Tier [Frontend UI]
        WebPage[หน้าเว็บร้านค้า Web Application]
    end

    %% Backend
    subgraph Backend_Tier [Backend API]
        APIServer[เซิร์ฟเวอร์ระบบ / Express.js Service]
    end

    %% Database & External Services
    subgraph Data_and_External_Tier [Database & Services]
        DB[(ฐานข้อมูลคลังสินค้า / MongoDB)]
        Payment[ระบบชำระเงิน / Payment Gateway]
    end

    %% Connections
    Customer -->|สั่งซื้อ/ชมสินค้า| WebPage
    Admin -->|จัดการสต็อก/ดูออเดอร์| WebPage
    WebPage -->|ส่งคำขอ HTTP Request| APIServer
    APIServer -->|บันทึกออเดอร์/ตัดสต็อก| DB
    APIServer -->|ตรวจสอบยอดเงิน| Payment
